💡 独特卖点 (vs 现有 Benchmark)

1 唯一覆盖 CUDA Kernel 的 SWE-bench 类 Benchmark

现有 SWE-bench / SWE-bench Pro / Multi-SWE 全部是纯 Python/JS/Java 项目。PaddleSWE-Bench 的 Track A 包含真实的 GPU kernel 修复（如 0-Size edge case、scatter assign 竞争条件），评测 Agent 对 C++/CUDA 代码的理解和修改能力。

2 RFC 驱动的结构化需求描述

每个 Feature 任务都有配套的 RFC 设计文档，包含 API 签名、参数说明、验收标准、参考实现。这天然等价于 SWE-bench Pro 中人工编写的 requirement specification，但我们无需额外标注——黑客松流程本身就要求提交 RFC。

3 来源于真实研发需求

数据不是人造 toy problem，而是飞桨框架的真实研发任务：API 缺失需要补全、算子精度需要修复、功能需要对齐 PyTorch。每个任务都经历了社区开发者实现 → CI 通过 → Reviewer 审核 → Merge 的完整流程，代表了工业级深度学习框架的真实复杂度。

4 数据污染风险极低

中文社区产出 + PaddlePaddle 生态（非 PyTorch/TensorFlow 主流训练语料）。黑客松任务描述在 community 仓库的 markdown 中，不会出现在常见的代码预训练数据集里。Agent 必须真正理解需求而非记忆答案。

5 任务类型分类 + 差异化评测

不像 SWE-bench 把所有任务混在一起报告单一 Pass@1。我们按 Bug Fix (Track A) 和 Feature Impl (Track C) 分别评测，避免"只会修 bug 不会写新功能"的 Agent 获得虚高分数。参考 SWE-Compass 的分类评测思路。

📝 样本示例

Track A: Bug Fix PaddlePaddle__Paddle-70102
BUG FIX paddle.nn.functional.fold

问题：fold 算子在特定输入 shape 下计算结果不正确

修复：修改 3 个文件（kernel + Python 接口），+24/-8 行，附带 14 行测试 patch

考察：Agent 需定位算子 bug 根因并修复，同时保证不引入 regression

Track A: Bug Fix PaddlePaddle__Paddle-74863
fix paddle.index_sample 0-Size

问题：当 batch_size/input_length/index_length 为 0 时 CUDA kernel crash

修复：在 GPU kernel 入口添加 0-size 提前 return，修改 2 个 .cu 文件，+9/-3 行

考察：Agent 需理解 CUDA kernel 的边界条件处理

Track C: Feature PaddlePaddle__Paddle-64519
为 Paddle 新增 cholesky_inverse API

需求：实现 paddle.linalg.cholesky_inverse，对称正定矩阵通过 Cholesky 分解求逆

实现：5 个文件，+277 行（Python 实现 + C++ kernel 注册 + 单测 + 文档）

RFC：community/pull/896 — 包含 API 签名、参数定义、与 PyTorch 对齐说明

考察：Agent 需根据 RFC 规范完整实现 API（前向/反向/注册/测试）

Track C: Feature PaddlePaddle__Paddle-64684
为 Paddle 新增 cartesian_prod API

需求：实现 paddle.cartesian_prod，计算多个一维张量的笛卡尔积

实现：7 个文件，+129 行（纯 Python 实现 + 单测）

RFC：community/pull/914

考察：相对简单的 Python-only API 实现，适合评测基础能力
